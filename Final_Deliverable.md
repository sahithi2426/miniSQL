# Final Deliverable: RDBMS Architecture Redesign

## 1. Critique of the Original System

The original MiniSQL system successfully implements the front-end of a database (Lexing, Parsing, AST generation, AST to Logical Plan translation) but radically diverges in its execution phase. Instead of passing the logical plan to a database execution engine, it uses **Ahead-of-Time (AOT) Query Compilation**. It converts the logical plan into a C program, embeds the table data directly as C structures and linked arrays within the source code, compiles it via `gcc`/`clang`, and executes it.

### Why This Divergence is Significant

While AOT compilation of queries (as seen in modern engines like HyPer) is a valid optimization technique for tight CPU loops, the original system applies it incorrectly by **compiling the storage layer into the query**.

- **Ephemeral Persistence:** Data only exists inside the arrays of the compiled C executable. When the program ends, the dataset vanishes.
- **Lack of Atomicity & Isolation:** Without an underlying storage abstraction, you cannot concurrently add rows or lock records.
- **Recompilation Overhead:** An `INSERT` query requires entirely rewriting the C source code and repeatedly triggering a heavy system-level compilation step.

### AOT Setup vs DBMS Interpreter

A real DBMS uses an **Interpreter-based Query Engine** running as a continuous background daemon. The engine maintains data on persistent storage (disk) and buffers it into memory. Queries are interpreted dynamically using modular operators traversing the data on-the-fly, allowing instant execution without touching a C compiler.

---

## 2. Redesigned Architecture

To transition to a standard relational database system comparable to PostgreSQL or SQLite, the system must separate concerns. We remove the `codegen` module entirely and introduce three isolated abstraction layers:

1. **The Storage Manager:** Handles interaction with the OS filesystem via fixed-size "Pages".
2. **The Buffer Pool Manager:** Sits between the Storage Manager and Execution. It pulls Pages from disk and pins them into finite RAM frames.
3. **The Execution Engine:** The modular runtime that traverses operations using data generated by the Buffer Pool.

### Why Separate Planning, Execution, and Storage?

Modularity is mandatory for scalability (Hellerstein et al.). The Execution Engine should only concern itself with relational algebra (Filtering, Joining). It should not care whether data is on an SSD platter, an NVMe drive, or cached in RAM. The Storage layer abstracts the I/O, removing disk lag from the algebraic logic.

---

## 3. Storage Abstraction & Data Access Strategy

Data can no longer be embedded into code; it must be mapped to persistent media without blowing up memory limits.

- **File and Page Architecture:** A table is persisted as a physical file on the OS. This file is segmented into fixed block chunks of 4KB or 8KB known as **Pages**.
- **Slotted Pages:** Within each Page, we use a Slotted Page layout. The page header contains pointers (slots) tracking the start and length of Tuples growing backward from the end of the page block. This manages variable-length strings efficiently.
- **The Buffer Pool:** A caching abstraction (Pavlo). When the Execution layer needs Tuples from "Page 4", it asks the Buffer Pool. The Pool loads Page 4 from the hard disk into an array of memory frames. Once the Execution layer reads the tuple, it "unpins" the page, allowing the Buffer Pool to evict it via an LRU-policy if memory runs out.

## 4. Reasoned Execution Model

### Chosen Execution Strategy

We transition from C-compiled arrays to the **Volcano Iterator Model** (Graefe, 1990). This is a **Pull-Based** system.

### Why Pull-Based?

While push-based architectures run faster on modern CPUs, they mirror the rigidity of AOT compilation. Pull-based iterators afford massive ease of implementation and elegant pipelining. Every physical operator (e.g., `SeqScan`, `Filter`, `Join`) exposes an identical interface: `init()`, `next()`, and `close()`.

### How Tuples Flow (The Pull-Based Pipeline)

In a pull-based model, data is actively requested by the root node and bubbles upwards.

1. A parent node (e.g., `LimitExec`) calls `next()` on its child (`FilterExec`).
2. The `FilterExec` calls `next()` on its child (`SeqScanExec`).
3. The `SeqScanExec` requests the current tuple from the Buffer Pool Manager.
4. The Buffer Pool fetches the physical page and passes the memory pointer back to `SeqScanExec`.
5. The Tuple evaluates up the tree. If `FilterExec` rejects it, `FilterExec` loops and calls `SeqScanExec.next()` again until a valid constraint is met, at which point it yields to `LimitExec`.

## 5. End-to-End Pipeline Mapping

We transition the existing compiler lifecycle to the runtime execution lifecycle:

| Stage               | Original (AOT) System                                                     | Redesigned (Standard DBMS) System                                                                                                |
| :------------------ | :------------------------------------------------------------------------ | :------------------------------------------------------------------------------------------------------------------------------- |
| **Parsing**         | SQL String → AST                                                          | SQL String → AST                                                                                                                 |
| **Analysis**        | Binder reads local python dict dictating schema.                          | Binder queries the **Persistent System Catalog** (itself stored in table Pages).                                                 |
| **Logical Plan**    | Tree of relational algebra (e.g. Scan -> Project)                         | Tree of relational algebra (e.g. Scan -> Project)                                                                                |
| **Physical Plan**   | N/A (Skipped entirely)                                                    | Planner determines optimal algorithms (e.g., Chooses `SeqScanExec` instead of `IndexScanExec` based on Cardinality estimates).   |
| **Code Generation** | `c_generator.py` writes static arrays into an isolated `.c` program file. | N/A (Removed. Code generation is discarded).                                                                                     |
| **Compilation**     | External OS call to `gcc/clang` to build executable.                      | N/A (Removed).                                                                                                                   |
| **Execution**       | `executor.py` runs compiled executable to `stdout`.                       | The root operator calls `root.next()`. The Volcano tree dynamically pulls tuples from the Buffer Pool and yields result streams. |

### Conclusion

By dropping C-code generation in favor of a Buffer-Pooled Volcano Iterator runtime, the MiniSQL system evolves from an inflexible compilation script into a theoretically sound, scalable, persistent Relational Database Management System.
